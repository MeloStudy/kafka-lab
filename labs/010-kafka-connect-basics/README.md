# LAB-010: Kafka Connect Basics

In this lab, we explore **Kafka Connect**, focusing on **Change Data Capture (CDC)** using **Debezium**.

Before proceeding, ensure you have read the theory in [CONCEPT.md](CONCEPT.md) regarding Connect architecture and the difference between polling and CDC.

## Infrastructure Dissection
We are using a `docker-compose.yml` that provisions:
- **Kafka**: KRaft-based broker (`apache/kafka:4.3.0`).
- **Schema Registry**: To manage schemas if we use Avro/Protobuf (currently optional, using JSON in this lab).
- **Kafka-UI**: For easy visualization of our topics and messages.
- **PostgreSQL**: A sample database configured with `wal_level=logical` to allow Debezium to read the replication log.
- **Kafka Connect**: Distributed worker container with the Debezium Postgres plugin pre-installed (`debezium/connect:2.5`).

## Step 1: Start Infrastructure

Start the environment:
```bash
docker-compose up -d
```
*Wait ~30 seconds for all containers, especially the `connect` worker, to start.*

## Step 2: Register the Source Connector

Kafka Connect in Distributed mode exposes a REST API. We will use `curl` to register our Debezium Postgres connector.

```bash
curl -X POST http://localhost:8083/connectors \
  -H "Content-Type: application/json" \
  -d '{
    "name": "pg-connector",
    "config": {
      "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
      "database.hostname": "postgres",
      "database.port": "5432",
      "database.user": "postgres",
      "database.password": "postgres",
      "database.dbname": "mydb",
      "topic.prefix": "pg",
      "plugin.name": "pgoutput"
    }
  }'
```

**Command Dissection:**
- `-X POST http://localhost:8083/connectors`: Submits a new connector configuration to the Connect REST API.
- `"connector.class": "io.debezium.connector.postgresql.PostgresConnector"`: The specific Java class inside the worker that implements the Postgres source logic.
- `"topic.prefix": "pg"`: All topics generated by this connector will start with `pg.`. For our `users` table, the topic will be `pg.public.users`.
- `"plugin.name": "pgoutput"`: Instructs Debezium to use the native Postgres logical decoding plugin.

Verify the connector is running:
```bash
curl -X GET http://localhost:8083/connectors/pg-connector/status
```

## Step 3: Trigger CDC Events

Connect to the PostgreSQL database and insert a new user. The `init.sql` script already created the table and some data, which Debezium captures via an initial snapshot. Let's create a live event.

```bash
docker exec -it <postgres-container-id> psql -U postgres -d mydb
```

In the `psql` prompt:
```sql
INSERT INTO users (first_name, last_name, email) VALUES ('Charlie', 'Brown', 'charlie@example.com');
UPDATE users SET last_name = 'Green' WHERE first_name = 'Alice';
DELETE FROM users WHERE first_name = 'Bob';
```

## Step 4: Verify in Kafka

Open **Kafka-UI** at `http://localhost:8080`.
Navigate to **Topics** -> `pg.public.users` -> **Messages**.

You will see:
1. The snapshot records (Alice, Bob).
2. The `INSERT` record for Charlie.
3. The `UPDATE` record for Alice.
4. The `DELETE` record for Bob. Notice that Debezium emits a message with a `null` payload (a tombstone) for deletes, which allows Kafka's log compaction to eventually clean up the history of that primary key.

## Self-Assessment
<details>
<summary>1. Why is tailing a transaction log (CDC) better than periodically polling a database table?</summary>
Polling adds load to the database, misses intermediate updates (if multiple happen between polls), and cannot detect hard deletes. CDC tails the transaction log asynchronously, capturing every single state change with minimal overhead, including deletes.
</details>

<details>
<summary>2. Where does Kafka Connect store its configuration and offset data in Distributed mode?</summary>
It stores them in internal Kafka topics (e.g., `connect_configs`, `connect_offsets`, `connect_statuses`). This is why it is fault-tolerant and stateless; if a worker dies, another worker can resume by reading the offsets from Kafka.
</details>

<details>
<summary>3. What is the role of a Converter in Kafka Connect?</summary>
A Converter translates the internal data representation used by Kafka Connect Tasks into the serialized format stored in Kafka (e.g., translating to Avro, JSON, or String) and vice versa.
</details>
